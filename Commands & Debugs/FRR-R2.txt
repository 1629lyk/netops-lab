BGP is perfect. B>* 0.0.0.0/0 via 10.0.12.2 is installed and active. The routing table is correct. The ping failure is not a routing problem — it's a forwarding/NAT problem on FRR-R2.

What's happening:
Packets from R1 reach R2 via the eBGP link (10.0.12.1 → 10.0.12.2), R2 forwards them out eth0 toward 8.8.8.8, but the reply comes back to R2 and R2 doesn't know to forward it back to R1. The return packet goes to VMware's NAT gateway (192.168.254.2) which has no route to 10.0.12.1. You need NAT masquerade on FRR-R2 so traffic from R1 appears to originate from R2's eth0 IP (192.168.254.135).

Fix — on FRR-R2 shell:
First install iptables:
bashapk add iptables
Then add NAT masquerade on the internet-facing interface:
bashiptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE
iptables -A FORWARD -i eth1 -o eth0 -j ACCEPT
iptables -A FORWARD -i eth0 -o eth1 -m state --state RELATED,ESTABLISHED -j ACCEPT
Enable forwarding if not already active:
bashsysctl -w net.ipv4.ip_forward=1
Make iptables rules persistent on Alpine:
bashapk add iptables-legacy
rc-update add iptables
/etc/init.d/iptables save
Make forwarding persistent:
bashecho 'net.ipv4.ip_forward=1' >> /etc/sysctl.conf

### ASA

Problem 1 — ASA can't ping 8.8.8.8
ASA reaches R1 (10.0.13.1 works) but internet fails. The ASA is sending packets out the outside interface toward R1, R1 forwards them to R2, R2 NATs them out — but the reply comes back to R2, gets forwarded to R1, R1 sends it back to ASA's outside IP (10.0.13.2).
The issue is R1 has no NAT for traffic originating from 10.0.13.0/24. Only traffic from R1 itself gets masqueraded by R2. Traffic from ASA (10.0.13.2) arrives at R2 still with source IP 10.0.13.2 — R2 NATs it, reply comes back to R2, but R2 doesn't know how to return it to 10.0.13.2 because that's on R1's eth2 which isn't in R2's routing table.
Fix — add the inside networks to R2's routing table so return traffic knows the path back:
On FRR-R2 console:
bashvtysh
configure terminal
ip route 10.0.13.0/30 10.0.12.1
ip route 10.0.14.0/24 10.0.12.1
ip route 10.10.0.0/24 10.0.12.1
ip route 10.20.0.0/24 10.0.12.1
end
write memory
This tells R2: for any return traffic destined to lab subnets, send it back via R1 (10.0.12.1). The iptables MASQUERADE on R2 then handles the NAT translation correctly for all these subnets.
