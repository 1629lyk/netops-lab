# Paste all commands and even debuggins

Bring up all interfaces:
baship link set eth0 up
ip link set eth1 up
ip link set eth2 up
Assign IPs:
baship addr add 192.168.66.101/24 dev eth0
ip addr add 10.0.12.1/30 dev eth1
ip addr add 10.0.13.1/30 dev eth2
Default route toward management (VMnet1 gateway):
baship route add 192.168.66.0/24 dev eth0
Enable forwarding:
bashecho 'net.ipv4.ip_forward=1' >> /etc/sysctl.conf
sysctl -p
Persistent interfaces:
bashcat > /etc/network/interfaces << 'EOF'
auto lo
iface lo inet loopback

auto eth0
iface eth0 inet static
    address 192.168.66.101
    netmask 255.255.255.0

auto eth1
iface eth1 inet static
    address 10.0.12.1
    netmask 255.255.255.252

auto eth2
iface eth2 inet static
    address 10.0.13.1
    netmask 255.255.255.252
EOF
Enable FRR daemons:
bashsed -i 's/bgpd=no/bgpd=yes/' /etc/frr/daemons
sed -i 's/ospfd=no/ospfd=yes/' /etc/frr/daemons
Write FRR config — BGP + OSPF with redistribution:
bashcat > /etc/frr/frr.conf << 'EOF'
frr version 8.2.2
frr defaults traditional
hostname FRR-R1
log syslog informational
no ipv6 forwarding
!
interface eth0
 description MGMT-TO-UBUNTU
 ip address 192.168.66.101/24
!
interface eth1
 description LINK-TO-FRR-R2
 ip address 10.0.12.1/30
!
interface eth2
 description LINK-TO-ASA-OUTSIDE
 ip address 10.0.13.1/30
!
router bgp 65001
 bgp router-id 10.0.12.1
 neighbor 10.0.12.2 remote-as 65002
 neighbor 10.0.12.2 description FRR-R2-ISP
 !
 address-family ipv4 unicast
  neighbor 10.0.12.2 activate
  redistribute ospf
 exit-address-family
!
router ospf
 ospf router-id 10.0.13.1
 redistribute bgp
 default-information originate always
 network 10.0.13.0/30 area 0
 network 192.168.66.0/24 area 0
!
ip route 192.168.66.0/24 eth0
!
line vty
!
EOF
Restart FRR:
bashrc-service frr restart
Checkpoint 2 — eBGP session and default route
bash# BGP session should come up within 30 seconds
vtysh -c "show bgp summary"
You should see neighbor 10.0.12.2 with state Established and a 1 in the PfxRcd column (the default route from R2).
bash# Default route should appear in routing table
vtysh -c "show ip route"
Look for B>* 0.0.0.0/0 [20/0] via 10.0.12.2 — the B means BGP learned, > means best path, * means installed in FIB.

FRR-R1 has not learnt the path to 0.0.0.0

Problem 1 — (Policy) blocking route exchange.
Both PfxRcd and PfxSnt show (Policy) which means FRR's default policy in version 8.x is denying all BGP prefixes. FRR 8.x ships with a deny-all default policy unlike older versions. You need to explicitly permit prefixes.
Problem 2 — No default route in the table.
R1 hasn't learned 0.0.0.0/0 from R2 because of Problem 1.

Fix — apply on FRR-R1 via vtysh:
vtysh
configure terminal
router bgp 65001
 address-family ipv4 unicast
  neighbor 10.0.12.2 route-map PERMIT-ALL in
  neighbor 10.0.12.2 route-map PERMIT-ALL out
 exit-address-family
!
route-map PERMIT-ALL permit 10
!
end
write memory

After applying on both, clear the BGP session to force re-advertisement:
vtysh -c "clear bgp * soft"
Wait 10 seconds then check on R1:
bashvtysh -c "show bgp summary"
vtysh -c "show ip route"
You should now see:

PfxRcd: 1 and PfxSnt with a number (not Policy)
B>* 0.0.0.0/0 [20/0] via 10.0.12.2 in the route table
